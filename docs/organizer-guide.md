# 主催者向けガイド（Admin Config Tool の使い方）

このガイドは、野外実習の主催者・教員が Admin Config Tool を使ってレイヤー構成やツアー（見学ポイント・ルート）を編集し、公開中のアプリへ反映するまでの手順を説明する。プログラミングの専門知識がなくても実施できるよう、コマンド操作は最小限に留めている。

## 前提: Admin Config Tool の起動方法

Admin Config Tool は参加者向けアプリとは別の、ローカルでのみ動かす管理用の Web ページである（Requirement 11.1）。GitHub Pages 上には公開していないため、手元のパソコンで以下の手順により起動する。

1. リポジトリを未取得の場合は clone する。
   ```
   git clone https://github.com/yokayoka/FieldTourMap.git
   cd FieldTourMap
   ```
2. 依存パッケージをインストールする（初回のみ）。
   ```
   npm install
   ```
3. ローカルサーバーを起動する。
   ```
   npm run dev
   ```
4. ターミナルに表示される URL（例: `http://localhost:5173/`）をブラウザで開き、以下のページへ移動する。
   - レイヤー編集: `http://localhost:5173/admin-tool/index.html`
   - ツアー編集（POI・ルート）: `http://localhost:5173/admin-tool/tour-editor.html`
   - 2 つのページ上部のナビゲーションリンクから互いに行き来できる。

開発環境の構築でつまずいた場合は [developer-guide.md](./developer-guide.md) の「ローカル開発環境構築」も参照。

## レイヤー編集（`admin-tool/index.html`）

地図タイル（地理院地図・シームレス地質図等）の追加・編集・削除を行う（Requirement 11.1〜11.5）。

1. ページを開くと、現在公開中の `layers.json` の内容が一覧表示される。
2. **新規追加**: 一覧の「追加」ボタンからフォームを開き、以下を入力する。
   - **ID**: 半角英数字のレイヤー識別子（既存レイヤーの編集時は変更不可）
   - **レイヤー名**: 画面に表示される名称
   - **種別**: 「ベース」（背景地図。同時に 1 枚だけ表示）または「オーバーレイ」（重ね表示。複数同時ON可）
   - **タイルURL**: `{z}/{x}/{y}` プレースホルダを含む XYZ形式のURL（例: `https://cyberjapandata.gsi.go.jp/xyz/std/{z}/{x}/{y}.png`）
   - **帰属表示（attribution）**: 地図の出典表記
   - **不透明度**: 0〜1（オーバーレイを重ねて見せたい場合は 0.5〜0.6 程度が目安）
   - **最小/最大ズーム**: タイル配信元が対応する範囲
   - **初期表示ON**: 参加者が最初にアプリを開いたときに表示するかどうか
3. 入力中、タイルURLの形式（`{z}/{x}/{y}` の有無等）や数値範囲が自動検証され、不正な場合はフォーム下部にエラーメッセージが表示され保存できない（Requirement 11.4）。
4. **プレビュー**: 「プレビュー」ボタンでフォーム右側の地図に実際のタイルを表示し、保存前に見た目を確認できる（Requirement 11.5）。
5. **編集・削除**: 一覧の各行から既存レイヤーを編集・削除できる。
6. 一通り編集が終わったら、画面上部の「layers.jsonをダウンロード」ボタンを押す。ブラウザのダウンロードフォルダに `layers.json` が保存される。

既存の `layers.json` を別ファイルから読み込みたい場合は、画面上部の「既存のlayers.jsonを読み込む」からファイルを選択する。

## ツアー編集（`admin-tool/tour-editor.html`）

見学ポイント（POI）・巡検ルート・メディアリンク・参考論文リンクを編集する（Requirement 4, 4.1, 4.2, 11）。

1. ページを開くと、現在公開中のサンプルツアーが初期状態として読み込まれる。
2. 画面上部で「ツアーID」「タイトル」を編集する。ツアーIDは `tours/<ID>.json` というファイル名になるため、半角英数字・ハイフンで指定する。
3. **POIを追加する**:
   1. 「POIを追加」ボタン（または一覧の「＋ 地図でPOIを追加」）を押してPOI追加モードにする。
   2. 地図上の見学ポイントの位置をタップ（クリック）する。
   3. 開いたフォームで名称・説明文を入力する。
   4. 「メディア」欄で写真・動画のリンク（Google Drive / Dropbox 等の共有URL）とキャプション、種別（写真/動画）を追加できる。行を追加/削除しながら複数件登録できる。
   5. 「参考文献」欄で参考論文等のURLと引用情報を同様に追加できる。
   6. 「保存」を押すと一覧・地図に反映される（この時点ではまだダウンロードされていない）。
4. **既存POIの編集・削除**: 地図上のマーカーをタップするか、一覧から選択して編集・削除する。
5. **ルートを追加する**:
   1. 「ルートを追加」ボタンを押してルート追加モードにする。
   2. 地図上で通過点を順にタップする（点線プレビューが表示される）。
   3. 2点以上タップしたら「ルートを確定」ボタンを押し、開いたフォームでルート名を入力して保存する。
6. 一通り編集が終わったら、画面上部の「tours/*.jsonをダウンロード」ボタンを押す。`<ツアーID>.json` という名前でダウンロードされる。

既知の制約: レイヤー選択（`layerIds`）の編集UI、およびルートの頂点の再編集（再描画）UIは未実装（読み込んだ値をそのまま保持する）。ルート名の変更のみ対応している。これらを変更したい場合は、ダウンロードしたJSONファイルをテキストエディタで直接編集する。

## ダウンロードしたJSONファイルの配置

ダウンロードしたファイルは、リポジトリの以下の場所に配置（上書き）する。

| ファイル | 配置先 |
| --- | --- |
| `layers.json` | `public/config/layers.json` |
| `<ツアーID>.json` | `public/config/tours/<ツアーID>.json` |

**新しいツアーを追加する場合**は、あわせて `public/config/tours/index.json` にも登録が必要（Task 20で追加した複数ツアー切替機能に対応するため）。

```json
[
  { "id": "sample-tour", "title": "サンプル巡検コース" },
  { "id": "second-tour", "title": "第二巡検コース（海岸地形）" },
  { "id": "<新しいツアーID>", "title": "<画面に表示するタイトル>" }
]
```

`index.json` に登録したツアーは、参加者向けアプリの画面左上「ツアーを選択」ボタンから切り替えられるようになる。一覧の先頭に登録したツアーが初期表示のツアーになる。

## Git へのコミット・push（変更を反映する）

配置・編集したファイルは、Git でコミットして GitHub にプッシュ（送信）することで初めて公開に反映される（Requirement 11.6, 11.7）。ターミナル（コマンドプロンプトやPowerShell）でリポジトリのフォルダに移動し、以下を実行する。

```
git status
```

変更したファイル（`public/config/layers.json` や `public/config/tours/*.json` 等）が表示されることを確認する。

```
git add public/config/layers.json public/config/tours/
git commit -m "レイヤー構成を更新（〇〇タイルを追加）"
git push
```

コミットメッセージは「何を変更したか」が分かるよう自由に記述してよい。Gitの操作に不慣れな場合は、コマンドの代わりに [GitHub Desktop](https://desktop.github.com/) 等のGUIツールを使ってもよい（変更ファイルを選択し、コミットメッセージを入力して「Commit」→「Push」するだけでよい）。

## 反映の確認

push すると GitHub Actions が自動的にビルド・デプロイを行う（数分程度）。

1. `https://github.com/yokayoka/FieldTourMap/actions` を開き、一番上のワークフロー実行が緑のチェックマーク（成功）になるまで待つ。
2. 成功したら `https://yokayoka.github.io/FieldTourMap/` を開き（キャッシュが残っている場合はスーパーリロード: `Ctrl+Shift+R` 等）、変更が反映されていることを確認する。
3. もしワークフローが赤色（失敗）になった場合、変更した JSON ファイルの形式に誤りがある可能性が高い。Actions のログの「Test」または「Type check」ステップの内容を確認するか、Admin Config Tool 側の検証エラーメッセージに従って修正し、再度コミット・pushする。

すべての変更履歴は Git のコミット履歴として残るため、過去の設定に戻したい場合は該当のコミットを確認・復元できる（Requirement 11.7）。

## Googleスプレッドシートでのプロジェクト作成（GitHubを使わない主催者向け、Requirement 16）

GitHubのアカウント作成やコマンド操作に抵抗がある主催者（他大学の教員等）は、Gitリポジトリを使わずに、Googleスプレッドシートだけで独自のツアーを作成し、参加者に公開できる。参加者向けアプリは、URLに `?project=<スプレッドシートID>` を付けてアクセスすると、Gitリポジトリのフォーク・デプロイ作業なしに、そのスプレッドシートの内容を読み込んで表示する。

### サンプルデータ

`docs/sample-project/` に、実際にアプリが読み込めるシート構成のサンプルをCSV形式で用意している（`Layers.csv`, `Tours.csv`, `POIs.csv`, `Media.csv`, `ReferencePapers.csv`, `Routes.csv`, `RoutePoints.csv` の7ファイル、公開中のサンプルツアーと同じ内容）。1つのGoogleスプレッドシートの中に、この7つのファイル名と同じ名前のシート（タブ）を作り、対応するCSVの内容をそれぞれ貼り付けることで、動作確認用のサンプルプロジェクトを再現できる。

1. 新しいGoogleスプレッドシートを作成する。
2. 下部のシートタブを7つ用意し、それぞれ `Layers`, `Tours`, `POIs`, `Media`, `ReferencePapers`, `Routes`, `RoutePoints` という名前に変更する（大文字・小文字も一致させる）。
3. 各CSVファイルの内容をテキストエディタで開いてコピーし、対応するシートのA1セルに貼り付ける（Googleスプレッドシートはカンマ区切りのテキストを自動的にセルへ展開する）。
4. 「ファイル」→「共有」→「ウェブに公開」から、スプレッドシート全体を公開する（詳細は次項）。

各シートの列構成は以下の通り（`tourId`/`poiId`/`routeId`は行同士を紐付けるキーで、1つのスプレッドシートに複数のツアーを同居させることもできる）。

| シート名 | 列 |
| --- | --- |
| `Layers` | id, name, type, urlTemplate, attribution, opacity, minZoom, maxZoom, defaultVisible |
| `Tours` | tourId, title, description, layerIds（カンマ区切り） |
| `POIs` | tourId, poiId, name, description, lat, lng |
| `Media` | tourId, poiId, url, caption, type（`photo`または`video`） |
| `ReferencePapers` | tourId, poiId, url, citation |
| `Routes` | tourId, routeId, name |
| `RoutePoints` | tourId, routeId, order（0始まりの通し番号）, lat, lng |

### スプレッドシートを「ウェブに公開」する

参加者向けアプリはOAuthやAPIキーを一切使わず、スプレッドシートの「ウェブに公開」機能で得られる公開URLを読み取るだけで動作する（ログイン不要・読み取り専用）。

1. Googleスプレッドシートを開き、「ファイル」→「共有」→「ウェブに公開」を選択する。
2. 「公開」ボタンを押す（既定の設定のままでよい）。
3. アドレスバーのURL（`https://docs.google.com/spreadsheets/d/<スプレッドシートID>/edit`）から `<スプレッドシートID>` の部分（`/d/`と`/edit`の間の文字列）を控える。
4. 参加者には `https://yokayoka.github.io/FieldTourMap/?project=<スプレッドシートID>` というURLを共有する。

**注意**: 「ウェブに公開」は誰でも閲覧できる状態になる設定である。個人情報や公開したくない情報を含めないこと。

### Admin Config Toolから直接保存・読み込みする（GitHubを使う運営者向け）

上記の手動でのシート作成・編集に加えて、Admin Config Tool（レイヤー編集・ツアー編集の両ページ）からOAuth経由でスプレッドシートへ直接保存・読み込みすることもできる（Requirement 15）。この機能を使うにはGoogle CloudでOAuthクライアントIDを発行する必要がある。

1. [Google Cloud Console](https://console.cloud.google.com/)で新しいプロジェクトを作成する（既存プロジェクトを流用してもよい）。
2. 「APIとサービス」→「有効なAPIとサービス」から「Google Sheets API」を検索して有効化する。
3. 「APIとサービス」→「OAuth同意画面」を設定する。個人利用や少人数の運営チームであれば、公開ステータスは「テスト」のままでよい。その場合、「テストユーザー」に自分（および共同運営者）のGoogleアカウントを追加する（未追加のアカウントでは認可に失敗する）。
4. 「APIとサービス」→「認証情報」→「認証情報を作成」→「OAuthクライアントID」を選択し、アプリケーションの種類は「ウェブアプリケーション」を選ぶ。「承認済みのJavaScript生成元」にAdmin Config Toolを配信するURL（例: `https://yokayoka.github.io`、ローカル確認時は`http://localhost:4173`等）を追加する。
5. 発行された「クライアントID」を、Admin Config Toolの「Googleスプレッドシート連携」欄の「OAuthクライアントID」に入力する（ブラウザのlocalStorageに保存され、次回以降は入力し直す必要はない）。
6. 「スプレッドシートID」欄に対象スプレッドシートのIDを入力し、「Googleでログイン」ボタンで認可する。認可後は「スプレッドシートに保存」「スプレッドシートから読み込む」ボタンが有効になる。
7. ツアー編集ページでは、画面上部の「ツアーID」欄に入力されているIDが保存・読み込みの対象になる。1つのスプレッドシートに複数のツアーを保存している場合、読み込み前に対象のツアーIDを入力しておくこと。

この機能は認可・読み書きにOAuthアクセストークンを使うAdmin Config Tool専用の機能であり、参加者向けMap Viewer（`?project=`パラメータでの読み込み）はOAuthを一切使わない読み取り専用のままである（Requirement 16）。

### 既知の制約

Admin Config Toolでの保存・読み込みが失敗しても、既存のJSONファイルのダウンロード・アップロード機能には影響しない。エラー時は画面上のステータス表示を確認し、必要に応じて上記の手動編集方法にフォールバックすること。

### 動作確認の範囲について

本機能（Requirement 15・16）は、単体テスト・E2Eテストにおいては実際のGoogleアカウント
やGoogle Cloud上のOAuthクライアントを使わず、Google Identity ServicesやSheets API・公開
CSVエンドポイントをすべてモック（模擬）した状態で動作確認している。そのため、以下は
このリポジトリの自動テストでは検証されておらず、実際にGoogle Cloud Console・Googleスプ
レッドシート上で設定した本人が手動で確認する必要がある。

- OAuth同意画面の実際の表示内容・テストユーザー制限の挙動
- 発行したOAuthクライアントIDが実際のドメイン（GitHub Pages配信ドメイン等）で正しく認可されること
- スプレッドシートの「ウェブに公開」設定が実際に反映され、参加者側から匿名で読み取れること
- Google側の仕様変更（API・GIS・公開CSVエンドポイントの形式変更等）による将来的な非互換

上記の設定を初めて行う際は、本番URLへ反映する前に、必ず自分自身のブラウザ・Googleアカウントで一連の操作（ログイン→保存→読み込み、および`?project=`URLでの参加者側表示）を一度試すこと。
