# Requirements Document

## Introduction
本プロジェクトは、地理学・地質学分野における大学等での野外教育（巡検・フィールドワーク）を支援するWebGISシステムを開発するものである。学生は野外実習中にスマートフォンのブラウザから本システムにアクセスし、GNSS（GPS等）による現在地情報と、地形図・地質図・空中写真などの複数の地図レイヤーを重ね合わせて閲覧することで、現地の地形・地質と地図情報を照合しながら学習することを目的とする。

本ドキュメントでは、要望文の曖昧な部分について以下の前提を置く。

- **主要ペルソナ**: (1) 野外実習に参加する学生（フィールド参加者）、(2) 実習を企画・引率する教員・TA（コンテンツ管理者）。
- **利用形態**: インストール不要のスマートフォン向けWebアプリ（レスポンシブWeb/PWA）とし、iOS/Android標準ブラウザで動作することを想定する。ネイティブアプリ化は本要件の対象外とする。
- **地図技術**: フロントエンドの地図描画にはLeafletを採用し、複数のタイルレイヤー（国土地理院地理院地図、産総研シームレス地質図、OpenStreetMap、空中写真等）をレイヤーコントロールで切り替え可能とする。
- **GNSS**: ブラウザのGeolocation APIを用いてスマートフォン端末のGNSS情報を取得し、地図上に現在地・進行方向・測位精度を表示する。
- **オフライン考慮**: 野外は通信環境が不安定な場所が多いことを想定し、事前に閲覧範囲のタイルをキャッシュしておくことで、電波が届かない状況でも直近にアクセスした範囲の地図閲覧が可能な設計を目指す（完全なオフライン新規タイル取得は対象外）。
- **教員機能**: 教員・TA（主催者）は、巡検ルートや観察地点（POI）・レイヤー構成・注記を事前に準備し、学生の端末に配信できることを想定する。
- **段階的リリース方針（重要な前提）**: 最初のバージョン（Phase 1）は、バックエンドサーバー・データベースを持たない**GitHub Pages上の静的コンテンツ**としてテスト運用することを目標とする。これに伴い、本ドキュメントにおける「主催者による登録・編集」は、次のような**Gitベース編集方式**を前提とする。
  - 主催者はローカル環境（または後述の管理用ローカルWebツール）でレイヤー構成・POI・ルート・写真動画リンク等をJSON設定ファイルとして編集する。
  - 編集したJSONファイルをGitリポジトリにコミット・プッシュ（または Pull Request をマージ）することで、GitHub Pages上の静的サイトに反映される。
  - そのため、Phase 1ではアプリ内へのログイン機能・サーバーAPI・データベースは設けず、リポジトリへの書き込み権限（GitHubアカウント権限）をもって「主催者権限」とみなす。
  - サーバーサイドでの動的なデータ保存、複数主催者による同時編集の排他制御、学生投稿データの中央集約などは、静的サイトの制約上Phase 1の対象外とし、将来のバックエンド導入フェーズ（Phase 2以降）で再検討する。
- **写真・動画の扱い**: 見学ポイント（POI）に添付する写真・動画は、当面は自前のメディアホスティングを持たず、Google DriveやDropbox等の外部共有リンクをJSON設定ファイル内にURLとして記述する運用とする。
- **ユーザー間コミュニケーションの切り分け**: 本システムはユーザー間の議論・コメント・チャット機能を意図的に持たない。ある地点の位置・表示内容を共有するための「URLによる表示状態の再現」機能のみを提供し、共有後の対話・議論はX（旧Twitter）やLINE等の外部SNS上で行うことを前提とする。これにより、ユーザー生成コンテンツ（UGC）のモデレーション等に伴う法的リスクを本システムが負わない設計とする。
- **データ永続化**: Phase 1ではルート・POI・レイヤー設定等はGitリポジトリ内のJSON設定ファイルとしてバージョン管理し、複数の実習・複数グループでの再利用を可能にする。学生個人の観察メモ等はブラウザのローカルストレージ（端末内）に保存し、サーバーへの集約はPhase 1の対象外とする。

## Requirements

### Requirement 1
**User Story:** As a 野外実習に参加する学生, I want スマートフォンのブラウザで地図上に自分の現在地を表示したい, so that 地図上の地形・地質と自分がいる実際の場所を照合しながら学習できる。

#### Acceptance Criteria
1. ユーザーが位置情報の利用を許可した場合、システムはGeolocation APIを用いて取得した現在地を地図上にマーカーで表示すること。
2. システムは位置情報の測位精度（誤差半径）を、地図上に円などで視覚的に表現すること。
3. 端末の向き（コンパス/方位センサー）が取得できる場合、現在地マーカーは進行方向・向きを示す表示に切り替わること。
4. 位置情報の利用が許可されない、または取得に失敗した場合、システムはエラーメッセージを表示し、地図の閲覧自体は継続して行えること。
5. 現在地の追従表示（自動センタリング）のON/OFFをユーザーが切り替えられること。

### Requirement 2
**User Story:** As a 野外実習に参加する学生, I want 地形図・地質図・空中写真など複数の地図レイヤーを切り替えたり重ねたりしたい, so that 地表の様子と地質構造を比較しながら観察できる。

#### Acceptance Criteria
1. システムは少なくとも3種類以上のベースレイヤー（例: 地理院地図の標準地図・写真、OpenStreetMap等）を提供すること。
2. ユーザーはレイヤーコントロールUIから表示するベースレイヤーを1つ選択して切り替えられること。
3. ユーザーはオーバーレイレイヤー（シームレス地質図、等高線、断層線、POI注記など）を複数同時にON/OFFできること。地質図はベースの地形図・空中写真と重ねて比較できるよう、半透明のオーバーレイレイヤーとして提供する。
4. レイヤー切り替え操作から2秒以内に表示が更新されること（キャッシュ済みタイルの場合）。
5. 選択中のレイヤー構成（表示中のベース・オーバーレイの組み合わせ）は、ページを再読み込みしても直前の状態を維持すること。

### Requirement 3
**User Story:** As a 野外実習に参加する学生, I want 通信が不安定な野外でも直近に見ていた範囲の地図を閲覧したい, so that 圏外・電波不良のフィールドでも実習を継続できる。

#### Acceptance Criteria
1. システムはService Worker等を用いて、閲覧済みのタイル画像をブラウザにキャッシュすること。
2. オフライン状態（通信不可）でも、キャッシュ済み範囲・ズームレベルの地図はエラーなく表示されること。
3. キャッシュされていないタイルへのリクエストがオフライン時に発生した場合、システムは代替表示（グレーアウト等）を行い、アプリ全体をクラッシュさせないこと。
4. 教員は事前に想定巡検エリアの地図タイルを一括ダウンロード（プリキャッシュ）できる機能を持つこと。

### Requirement 4
**User Story:** As a 実習を企画する教員・主催者, I want 巡検ルートや観察地点（POI）を事前に登録し学生の地図に配信したい, so that 学生が迷わず観察ポイントを回れるようにしたい。

#### Acceptance Criteria
1. 主催者は、管理用ローカルツール（または地図上をクリック/タップする編集UI）を用いて観察地点（POI）の位置・属性を作成し、JSON設定ファイルとして出力できること。
2. POIには名称、短い説明文（テキスト）を必須項目として付与できること。
3. 主催者は複数のPOIを結んだ巡検ルート（線データ）を作成し、同様にJSON設定ファイルとして出力できること。
4. 出力されたJSON設定ファイルをGitリポジトリにコミット・プッシュすることで、学生側の地図に自動的にオーバーレイレイヤーとして表示されること。
5. 主催者は作成したルート・POIセットを実習単位（コース・回次）ごとにファイル分割し、保存・再利用できること。

### Requirement 4.1
**User Story:** As a 実習を企画する教員・主催者, I want 見学ポイントに写真や動画のリンクとそれらに関する短い説明文を追加したい, so that 学生が現地で該当地点の参考資料（過去の写真・解説動画等）を確認しながら観察できる。

#### Acceptance Criteria
1. 主催者はPOIごとに、写真または動画への外部リンク（URL）を1件以上添付できること。
2. 添付リンクは当面Google DriveまたはDropboxの共有リンク形式を想定し、システムはリンク先のファイルを自前でホスティング・変換しないこと。
3. 各リンクには、その写真・動画の内容を説明する短いテキスト（キャプション）を付与できること。
4. 学生側の地図でPOIをタップすると、名称・説明文に加えて、添付された写真・動画リンクとキャプションの一覧が表示されること。
5. 学生が写真・動画リンクをタップすると、新しいタブ（またはリンク先アプリ）でGoogle Drive/Dropbox上のコンテンツが開くこと。システムはリンク切れの場合でも他の情報表示に影響を与えないこと。
6. システムはリンクURLの形式（`http(s)://`で始まるか等）を簡易検証し、明らかに不正な値が設定ファイルに含まれる場合は、ビルド時またはロード時に警告を出せること。

### Requirement 4.2
**User Story:** As a 実習を企画する教員・主催者, I want 見学ポイントに関連する参考論文（公開されているPDFへのリンク）を追加したい, so that 学生が現地でその地点の学術的背景・出典を確認しながら観察できる。

#### Acceptance Criteria
1. 主催者はPOIごとに、公開されている論文PDFへの外部リンク（URL）を0件以上添付できること。
2. 参考論文リンクは、Google Drive/DropboxのようなファイルホスティングサービスだけでなくJ-STAGE、DOIリンク、大学・学会リポジトリ等、任意の公開URLを許容すること。システムはリンク先のPDFを自前でホスティング・変換しないこと。
3. 各参考論文リンクには、論文タイトルまたは出典（著者・発行年等を含む短い引用表記）を付与できること。
4. 学生側の地図でPOIをタップした際、写真・動画リンクとは区別できる形（例: 「参考文献」セクション）で参考論文リンクの一覧が表示されること。
5. 学生が参考論文リンクをタップすると、新しいタブでリンク先のPDF/ページが開くこと。リンク切れの場合でも他の情報表示に影響を与えないこと。
6. システムはリンクURLの形式を簡易検証し、明らかに不正な値が設定ファイルに含まれる場合は、ビルド時またはロード時に警告を出せること。

### Requirement 5
**User Story:** As a 野外実習に参加する学生, I want 観察した内容を現在地に紐づけて記録したい, so that 実習後にレポート作成や振り返りに活用できる。

#### Acceptance Criteria
1. 学生は現在地または任意の地図上の地点に、テキストメモを追加できること。
2. 記録した観察メモは地図上にピンとして表示され、タップすると内容を確認できること。
3. 記録したメモはPhase 1では端末のブラウザローカルストレージ（localStorage/IndexedDB等）に保存され、同一端末・同一ブラウザでのセッションをまたいで失われないこと。サーバーへの集約保存は将来フェーズの対象とする。
4. 学生は自分が記録したメモの一覧をエクスポート（CSV/GeoJSON等）できること。

### Requirement 6
**User Story:** As a システム利用者（学生・教員共通）, I want どのスマートフォン画面サイズでも快適に操作したい, so that 野外の片手操作でもストレスなく地図を確認できる。

#### Acceptance Criteria
1. UIはスマートフォンの縦持ち画面（幅360px程度）を基準にレスポンシブ対応すること。
2. 主要な操作（現在地表示、レイヤー切替）は片手の親指操作で到達可能な位置に配置すること。
3. 屋外の強い日差し下でも視認できるよう、十分なコントラスト比を確保すること。
4. タップ対象は44px角以上を確保し、誤操作を防止すること。

### Requirement 7
**User Story:** As a システム管理者, I want 地図タイル配信や位置情報取得のパフォーマンスを一定水準に保ちたい, so that 大人数の実習利用でもシステムが安定して動作する。

#### Acceptance Criteria
1. 初回ロード時、地図の初期表示（現在地周辺）が3秒以内に完了すること（良好な通信環境下）。
2. 位置情報の更新間隔は1〜5秒程度とし、バッテリー消費と精度のバランスを考慮した設定を提供すること。
3. Phase 1ではアプリ本体・設定ファイルはGitHub Pages（静的CDN配信）でホスティングされ、アプリ自体のオリジンサーバー負荷は考慮不要とする。ただし外部タイル配信元（国土地理院等）へのリクエスト過多を避けるため、クライアント側キャッシュを活用すること。
4. 将来的にサーバーサイド機能（Phase 2以降）を追加する場合、同時アクセス50セッション程度までレスポンスタイムが著しく劣化しないことを設計上の目標とする。

### Requirement 8
**User Story:** As a システム利用者, I want 自分の位置情報や記録した観察データが適切に保護されている, so that 個人の行動履歴が不用意に外部へ漏洩しない。

#### Acceptance Criteria
1. サイトはGitHub Pagesの標準機能によりHTTPSで配信されること。
2. 学生の位置情報・観察メモはPhase 1では端末内（ローカルストレージ）にのみ保存され、外部（他の学生・第三者）に自動送信・共有されないこと。
3. 主催者向けの認証情報管理（アカウント・パスワード等）はPhase 1では不要とし、GitHubリポジトリへの書き込み権限をアクセス制御の代替手段とする。将来サーバーサイド機能を導入する際は、認証情報を安全に（ハッシュ化等）保存すること。
4. 個人を特定しうる位置履歴データを将来的にサーバー保存する場合は、実習終了後一定期間で削除またはアーカイブ可能なポリシーを持つこと。

### Requirement 9
**User Story:** As a 開発者, I want システムが自動テストによって品質担保されている, so that 機能追加や修正の際にデグレードを早期に検知できる。

#### Acceptance Criteria
1. GNSS位置表示、レイヤー切替、POI登録などの主要機能について単体テストが存在すること。
2. 地図タイル配信やAPIエンドポイントについて結合テストが存在すること。
3. CI環境でテストが自動実行され、失敗時にビルドが失敗すること。
4. コードは主要な関数・コンポーネント単位でドキュメントコメントまたは型定義により保守性が確保されていること。

### Requirement 10
**User Story:** As a 開発者・運用担当者, I want 将来的な地図レイヤーの追加や設定変更が容易である, so that 新しい調査対象地域や新しい地質図データにも柔軟に対応できる。

#### Acceptance Criteria
1. 地図レイヤーの定義（タイルURL、名称、種別、初期表示ON/OFF等）は設定ファイル（JSON等）で管理され、コード変更なしに追加・変更できること。
2. 新しいタイルレイヤーソース（WMS/XYZ形式等）を追加する際、既存コードへの影響を最小限にできる設計であること。

### Requirement 11
**User Story:** As a 実習を企画する教員・主催者, I want タイルマップのアドレス（URL）を指定してWebGISのレイヤー構成を編集したい, so that アプリ本体のコードを変更せずに、実習に応じた地図レイヤーを自由に追加・修正・削除できる。

#### Acceptance Criteria
1. 主催者は、管理用ローカルWebツール（アプリ本体とは別に配布されるローカル編集画面、またはリポジトリ内のフォーム付きHTML）から、現在登録されているレイヤー一覧（ベース/オーバーレイ）を閲覧・編集できること。
2. 主催者はタイルURL（XYZ形式のURLテンプレート、例: `https://.../{z}/{x}/{y}.png`）、レイヤー名、レイヤー種別（ベース/オーバーレイ）、初期表示ON/OFF、不透明度、最小/最大ズームレベル、帰属表示（attribution）を入力し、レイヤー定義をJSON設定ファイルとして追加・編集できること。
3. 主催者は不要になったレイヤー定義をJSON設定ファイルから削除できること。
4. システムはレイヤー追加・編集時に、指定されたタイルURLの形式（`{z}/{x}/{y}`プレースホルダの有無等）を検証し、不正な形式の場合はエラーメッセージを表示して保存（出力）を拒否すること。
5. 主催者は編集中のレイヤー設定を保存前に、管理用ツール上の地図でプレビュー表示できること。
6. レイヤー構成の変更は、JSON設定ファイルをGitリポジトリにコミット・プッシュしGitHub Pagesへの再デプロイが完了した後、学生側の地図画面に反映されること。
7. レイヤー構成データはGitリポジトリ内のJSON設定ファイルとしてバージョン管理され、変更履歴（git履歴）が失われないこと。

### Requirement 12
**User Story:** As a 開発者・運用担当者, I want 最初のバージョンをGitHub Pagesの静的コンテンツとしてデプロイ・テスト運用したい, so that サーバー運用コストをかけずに早期に実利用者からフィードバックを得られる。

#### Acceptance Criteria
1. アプリケーションはビルド後、サーバーサイド処理を必要としない静的ファイル一式（HTML/CSS/JS/JSON設定ファイル）として出力されること。
2. 出力された静的ファイル一式は、GitHub Pages（`main`ブランチまたは`gh-pages`ブランチ、もしくは`/docs`フォルダ配信）にデプロイして正常に動作すること。
3. GitHub ActionsなどのCIにより、リポジトリへのプッシュ時に静的ファイルのビルドとGitHub Pagesへの自動デプロイが行われること。
4. ローカル開発環境において、GitHub Pagesと同一の相対パス構成でアプリを起動・検証できること（サブパス配信を考慮したbase pathの設定等）。
5. 本要件はPhase 1（MVP）のデプロイ先をGitHub Pagesに限定するものであり、将来的にサーバーサイド機能（Requirement 5, 7, 8の一部）を追加する際は、別途ホスティング先の見直しを行うことを妨げない。

### Requirement 13
**User Story:** As a 野外実習に参加するユーザー, I want 任意のポイントの位置と表示コンテンツを他のユーザーと共有したい, so that 共有相手が自分の見ている場所・ズームレベル・表示レイヤーをそのまま再現でき、議論自体はSNS等の使い慣れたツールで行える。

#### Acceptance Criteria
1. ユーザーは現在表示中の地図ビュー（中心座標、ズームレベル、表示中のベースレイヤーおよびオーバーレイレイヤーの組み合わせ）を一意に表す共有URLを生成できること。
2. 特定のPOI（見学ポイント）の詳細情報パネルを開いた状態で共有した場合、共有URLにはそのPOIの識別子が含まれ、受信者側でも同じPOI詳細パネル（写真・動画リンク・説明文を含む）が自動的に開いた状態で再現されること。
3. 生成された共有URLは、クリップボードへのコピー操作、または対応端末ではWeb Share API（OSネイティブの共有シート）を通じて、X・LINE等の外部アプリに直接渡せること。
4. 共有URLを受け取ったユーザーが当該URLにアクセスした場合、サーバーへの問い合わせなしに、URL自体にエンコードされた情報のみから送信者が見ていた地図の中心・ズームレベル・レイヤー構成・（該当する場合）POI詳細を再現できること（静的サイトの制約上、共有状態はURLのクエリパラメータまたはハッシュにのみエンコードし、サーバー側には保存しないこと）。
5. システムは共有相手との間でコメント・チャット・返信・リアクション等、対話のためのUI機能をアプリ内に実装しないこと。ユーザー間の議論は共有URLをX・LINE等の外部SNS上でやり取りすることで行う設計とし、本システムはコンテンツ提供とビュー状態の共有のみに責務を限定すること。
6. 共有URLの状態エンコードは、主要なSNS・メッセージアプリでの共有時に問題とならない長さに収まるよう、簡潔な形式（短いクエリパラメータ名、必要に応じたエンコード・圧縮）を用いること。
7. 不正・破損した共有URL（パラメータ欠損、範囲外の値等）を受け取った場合、システムはエラー画面で停止せず、アプリの初期表示（デフォルトビュー）にフォールバックすること。

### Requirement 14
**User Story:** As a 車で巡検（ツアー）を行う参加メンバー, I want 任意のポイントをGoogleマップで開けるリンクをクリップボード経由で取得したい, so that 自分のGoogleマップアプリで経路検索を行ったり、SNSで他のメンバーと共有したりして、各ポイントへの移動ルート検討を補助できる。

#### Acceptance Criteria
1. ユーザーは地図上の任意の地点（現在地、POI、地図上で任意にタップした地点等）に対し、「Googleマップで開くリンクを取得」操作を実行できること。
2. システムは選択地点の緯度経度から、Googleマップのピン留め（検索）形式のURL（例: `https://www.google.com/maps/search/?api=1&query={lat},{lng}`）を生成すること。この形式を採用するのは、送信先が自分のGoogleマップアプリでも他者へのSNS共有でも、出発地に依存せず汎用的に使えるためである（Requirement 13のアプリ内ビュー共有とは異なり、外部サービスであるGoogleマップの標準リンク形式に従う）。
3. 生成したリンクはクリップボードにコピーできること。対応端末ではWeb Share APIにも対応し、X・LINE等への直接共有も可能とすること。
4. コピーしたリンクをGoogleマップアプリまたはブラウザで開いた場合、指定した地点が正しい位置にピン表示されること。
5. クリップボードへのコピーが成功した際、ユーザーにコピー完了を示す簡潔なフィードバック（トースト通知等）を表示すること。
6. クリップボードAPIが利用できない環境（非HTTPS、非対応ブラウザ等）では、リンク文字列を選択・手動コピーできる代替UI（テキスト表示）にフォールバックすること。
